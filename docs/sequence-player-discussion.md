# 序列脚本播放器 — 讨论思考记录

> 日期：2026-04-02
> 参与者：用户（产品方）+ AI（技术方）

---

## 1. 需求起源

### 1.1 背景

现有课程制作流程依赖传统视频录制：拍摄/录屏 → 剪辑 → 导出视频。这个流程存在明显痛点：
- 视频制作门槛高（需要剪辑技能）
- 修改成本高（改一句话可能需要重新录制整个视频）
- 素材无法复用（角色、背景等每次都要重新做）

### 1.2 灵感来源

用户观察到，现有课程视频的画面结构与 Galgame（视觉小说）引擎高度相似：

| 课程视频元素 | Galgame 对应概念 |
|------------|-----------------|
| 场景背景（工作坊/教室） | 背景切换系统 |
| 蓝色边框内的展示画面 | CG 展示区 |
| 右下角卡通讲师角色 | 角色立绘 + 表情系统 |
| 思维气泡（玩具等） | 叠加元素/特效层 |
| 底部台词字幕 | 对话框 + 字幕系统 |

→ 结论：完全可以用引擎实时渲染替代视频文件。

### 1.3 核心目标

> **只要导入逐字稿，就能自动生成可播放的课程序列。**
> 
> 不再需要制作视频，只要制作序列事件。

---

## 2. 第一轮讨论：架构方向

### 2.1 初始方案（v1）— 精细化事件编排

最初设计了一个完整的事件系统，包含 20+ 种事件类型：
- `dialogue`, `narration`, `show_image`, `show_video`
- `character_enter`, `character_exit`, `character_emotion`
- `bg_change`, `bgm_play`, `sfx_play`, `transition`
- `choice`, `quiz`, `code_demo`, `scratch_demo` ...

每个 Step 包含多个并行 Event，类似专业 Galgame 引擎的事件编排。

### 2.2 用户反馈 — 方向转折

用户明确指出：

> "不要着急确认那么多事件的类型，我希望这个的作用是快速产出播放产物，而不是要精细化操作。我最终的目标是，只要导入指定的逐字稿，就会自动拆分出来怎么播放。"

**关键洞察**：
- 用户不需要一个"编辑器"，需要的是一个"自动化流水线"
- 目标用户是教研老师，不是技术人员
- 速度 > 精细度
- AI 编排 > 手动编排

### 2.3 方案修正（v2）— AI 自动编排

核心思路转变：

| | v1 | v2 |
|--|--|--|
| 理念 | 手动编排每个事件 | 逐字稿 → AI 自动处理 |
| 数据模型 | Step 包含多个 Event | Step = 台词 + 表情 + 展示 |
| 编辑器 | 复杂时间轴 | 简单微调面板 |
| 目标 | 精细控制 | **3步出片** |

## 3. 第二轮讨论：关键决策确认

### Q1: 角色立绘来源

**确认：已有现成素材，直接导入使用。**

不需要 AI 生成角色，已有课程 IP 角色（如截图中的卡通男孩讲师）。需要设计一个素材导入和管理的机制，让老师把已有的角色表情组图片上传到系统中。

### Q2: 展示画面处理

**确认：需要有一定的序列编排机制，具体方案后续讨论。**

展示画面不是简单的"台词旁边放张图"，而是需要配合台词节奏来编排展示内容的切换。这个机制的具体设计留待后续讨论，可能涉及：
- 逐字稿中标注展示提示 → AI 自动匹配
- 一个时间轴式的展示内容编排功能
- 展示素材与台词步骤的关联方式

### Q3: TTS 声音策略

**确认：角色组制，声音与立绘配套。**

每个角色是一个完整的"角色组"概念：
- 角色名
- 立绘表情组（多张图片）
- TTS 声线 ID（与 TTS Studio 的声音库对接）
- 默认位置

创建课程时选择主讲角色，该角色的所有属性（外观+声音）自动配套使用。

### Q4: 播放场景

**确认：按学生观看视角设计。**

虽然短期内只是老师在 Nautilus 上预览自查，但播放器的 UI 和交互要按学生实际观看的体验标准来设计：
- 沉浸式全屏体验
- 友好的播放控件
- 流畅的动画过渡
- 清晰的字幕显示

这样保证做出来的东西就是最终产物的真实预览。

## 4. 当前确定的技术架构

### 4.1 模块定位

| 属性 | 值 |
|------|---|
| 模块名 | `sequence-player` |
| 平台 | 仅 Nautilus |
| 前端 | Vue3 + NaiveUI + CSS Animations |
| 后端 | Express + better-sqlite3 |
| AI | Gemini API（逐字稿 → 序列脚本） |
| TTS | 复用 TTS Studio 服务 |

### 4.2 播放器渲染层

```
Layer 5: UI 控制层（进度条、菜单按钮）
Layer 4: 字幕层（底部台词条 + 打字效果）
Layer 3: 角色层（讲师角色 + 情绪表情切换）
Layer 2: 内容展示层（蓝边框内的图片/视频）
Layer 1: 叠加元素层（思维气泡、标注）
Layer 0: 背景层（场景背景图）
```

### 4.3 核心数据流

```
逐字稿文本
    ↓ [Gemini API]
序列脚本 JSON（场景→步骤→台词/表情/展示）
    ↓ [TTS Studio API]
音频文件生成
    ↓
播放器渲染
```

## 5. 待后续讨论的问题

### 5.1 展示画面的序列编排机制

这是最需要深入讨论的部分。课程视频中，「内容展示区」的画面不是随便放的，它需要配合台词的节奏来切换：
- 老师说"看这台3D打印机"→ 画面展示 3D 打印机
- 老师说"它一层一层地打印"→ 画面可能切到打印过程的特写
- 老师说"可以打印出玩具"→ 画面切到打印出来的成品

需要设计一个既简单又能满足这种需求的编排方式。

### 5.2 多角色对话

目前先做单角色（主讲老师），后续可能需要支持：
- 主讲 + 助教对话
- 角色扮演场景

### 5.3 课程包管理

如何管理已创建的课程序列：
- 列表管理
- 版本迭代
- 导出/分享

## 6. 开发计划

### Phase 1 — 播放器核心 + AI 编排（优先）
1. 创建 `sequence-player` 模块骨架
2. 实现播放器渲染引擎（6 层渲染）
3. 实现 AI 逐字稿解析 → 序列脚本生成
4. 实现 TTS 批量语音生成
5. 基础播放控制

### Phase 2 — 编辑面板 + 微调
1. 逐字稿导入界面
2. 步骤列表 + 可视化微调
3. 素材管理（角色组导入）
4. 预览 + 发布流程

### Phase 3 — 展示编排增强
1. 展示画面的序列编排机制（待讨论后实现）
2. 更丰富的动画效果
3. 多角色支持
